# 图片生成双链路改造设计

日期：2026-05-14

## 背景

当前项目的图片生成功能前端和协议层默认使用 `gpt-image-2`，但后端最终会把该模型映射为 `gpt-5-3`，并通过 ChatGPT Web 的 conversation 图片链路发起请求。这条链路可以出图，但存在几个明显问题：

1. 实际调用模型与用户认知不一致，排障成本高。
2. `size` 主要通过提示词附加说明约束，不是原生图片参数控制。
3. 缺少稳定的质量档位和风格参数控制。
4. 当上游行为波动时，当前链路无法区分“高质量默认链路失败”和“历史兼容链路回退成功”。

用户目标是：

- 默认优先走更接近原生高质量生图的链路。
- 保留当前 conversation 图片链路作为回退。
- 尽量不改前端接口和交互。
- 让线上行为可观测、可回退。

## 目标

本次改造的目标：

1. 建立图片生成后端适配层，显式区分“默认高质量链路”和“兼容回退链路”。
2. 保持现有前端请求格式兼容：`prompt`、`model`、`n`、`size`、编辑参考图继续沿用。
3. 去掉当前隐式的 `gpt-image-2 -> gpt-5-3` 直写映射，把模型选择和链路选择放到适配层。
4. 增加配置开关，允许默认走高质量链路，并在失败时自动回退。
5. 增加日志和测试，能明确看出每次请求实际命中了哪条链路。

不在本次范围内的内容：

- 前端大规模 UI 重构。
- 全量替换或删除现有 conversation 图片链路。
- 新增复杂的图片风格面板或高级参数面板。

## 方案对比

### 方案 A：仅修正模型映射

只去掉 `gpt-image-2 -> gpt-5-3` 的硬编码映射，其余链路不动。

优点：

- 风险最小。
- 改动量小。

缺点：

- 仍然走现有 conversation 图片链路。
- 质量控制问题基本不变。
- 无法满足“默认高质量，失败回退”的目标。

### 方案 B：双链路适配层

新增默认高质量图片链路，并保留当前 conversation 图片链路作为兼容回退。

优点：

- 满足默认高质量和可回退的要求。
- 不需要改前端接口。
- 可逐步上线，风险可控。

缺点：

- 后端抽象和测试量会增加。

### 方案 C：全量替换旧链路

直接废弃 conversation 图片链路，只保留新链路。

优点：

- 长期代码最干净。

缺点：

- 没有回退余地，不适合当前线上状态。

推荐方案：**方案 B**

## 设计

### 1. 新增图片后端适配层

新增一层统一入口，负责根据配置和运行结果选择具体图片后端。接口保持统一，至少覆盖：

- 文生图
- 图生图/编辑
- 批量 `n` 张
- 比例/尺寸传递

建议结构：

- `services/image_backends/base.py`
- `services/image_backends/native.py`
- `services/image_backends/conversation.py`

统一能力：

- `generate(...)`
- `edit(...)`
- 返回统一结果结构，供现有 `conversation.py` / 协议层继续消费。

### 2. 默认高质量链路

`native` 后端作为默认路径，负责：

- 显式接收 `model`
- 尽可能用原生参数传递尺寸/比例，而不是仅拼接提示词
- 保留必要的鉴权和账号池逻辑

这里的“原生”不是要求前端改协议，而是要求后端内部不要再把图片生成单纯当作一段聊天消息拼给 conversation 链路处理。

### 3. 兼容回退链路

当前 `conversation` 图片链路保留，但职责改为：

- 作为 fallback
- 兼容历史账号能力差异
- 兼容某些新链路失败时的降级

它不再承担默认高质量职责。

### 4. 路由与模型选择逻辑

新增配置项，建议放入 `config.json` / `services/config.py`：

- `image_backend_default`: `native` | `conversation`
- `image_backend_fallback_enabled`: `true` | `false`
- `image_backend_fallback_order`: 固定为 `["native", "conversation"]`，当前只需支持这个顺序

运行逻辑：

1. 收到图片请求
2. 先按默认后端执行
3. 若触发可回退错误，则自动切到下一个后端
4. 若命中不可回退错误，则直接报错

### 5. 回退判定

可回退错误包括：

- 上游连接错误
- 新链路明确返回“不支持该能力/模型”
- 新链路解析失败但不是用户参数错误

不可回退错误包括：

- prompt 为空
- 图片编辑缺少参考图
- 参数格式非法
- 明确的鉴权失败或额度不足（避免回退掩盖真实问题）

### 6. 日志与可观测性

每次图片请求至少记录：

- 请求模式：`generate` / `edit`
- 请求模型名
- 请求尺寸
- 默认后端
- 实际命中后端
- 是否发生回退
- 回退原因

目标是让线上日志能回答两个问题：

1. 当前这张图到底走了哪条链路？
2. 质量差是默认链路产物，还是回退链路产物？

- 协议层入口 `openai_v1_image_generations.py` / `openai_v1_image_edit.py` 继续维持现有签名

  - [services/openai_backend_api.py](/Users/penglinchang/Desktop/code/chatgpt2api/services/openai_backend_api.py)
  - 新增 `services/image_backends/*`

### 8. 测试策略

至少覆盖以下场景：

1. 默认 `native` 成功，不触发回退
2. 默认 `native` 失败，自动回退 `conversation` 成功
3. 默认 `native` 失败，回退关闭时正确报错
4. 参数错误不触发回退
5. `model`、`size`、`n` 在两条链路中都能正确传递
6. 编辑模式参考图传递正确

## 实施顺序

1. 抽出图片后端统一接口
2. 把现有 conversation 图片能力迁移成 `conversation` 后端
3. 接入 `native` 后端
4. 在统一入口加入默认链路 + fallback 逻辑
5. 加日志
6. 补测试
7. 最后再决定是否需要前端暴露“当前走了哪条链路”的调试信息

## 风险与控制

风险：

- 新链路和旧链路的返回结构不完全一致
- 某些账号仅支持旧链路
- 回退条件定义不准时，可能掩盖真实错误

控制方式：

- 统一结果结构后再进入现有协议层
- 默认开启回退，但只对明确可回退错误生效
- 日志必须记录默认后端、实际后端和回退原因

## 验收标准

满足以下条件即可认为本次改造完成：

1. 不改前端接口即可正常出图
2. 默认请求优先走高质量后端
3. 新链路失败时可自动回退旧链路
4. 日志中能明确看到实际链路
5. 现有图片任务接口和前端页面不出现回归
